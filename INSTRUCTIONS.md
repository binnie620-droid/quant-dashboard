# UCHIDA V3 — AI 협업 지시사항

이 문서는 UCHIDA 프로젝트에서 AI(Claude 등)와 협업할 때 반드시 지켜야 할 원칙을 정의한다. 매 세션 시작 시 AI에게 이 파일을 읽도록 한다.

---

## 1. 절대 원칙

### 1-1. 근거의 우선순위

모든 주장과 결정에는 근거가 필요. 우선순위:

1. **학계 논문** (peer-reviewed journals)
   - 예: *Journal of Finance*, *Review of Financial Studies*, *Journal of Financial Economics*, *Econometrica*
   - 저자, 연도, 저널명 명시
2. **업계 표준** (공인 기관)
   - 예: Vanguard 백서, AQR 연구, Bloomberg 데이터
3. **데이터 실증** (우리 백테스트)
   - 명시적으로 "백테스트 결과"라고 표시
4. **AI 자체 판단**
   - **반드시 명시할 것**: "AI 판단" 또는 "구체적 수치는 AI 추정"
   - 학계 권장 범위 내에서만 판단

### 1-2. 가짜 논문 절대 금지

**가장 중요한 원칙**:

- 존재하지 않는 논문 인용 금지
- 저자, 연도, 제목 중 하나라도 불확실하면 **그렇게 명시**
- "학계에 그렇게 알려졌다" 정도면 OK (구체적 인용 안 해도 됨)
- **불확실하면 "학계 일반 견해" 또는 "정확한 인용 미확인"이라고 표시**

좋은 예시:
```
✓ "Fama & French (1992), 'The Cross-Section of Expected Stock Returns', JoF"
✓ "학계 일반 견해로는 X"
✓ "VIX 임계값에 대한 합의된 논문은 못 찾았음"
```

나쁜 예시:
```
✗ "Johnson et al. (2019)" (존재 확인 안 됨)
✗ 가짜 논문 제목 만들어내기
✗ 추정값을 학계 컨센서스인 것처럼 표현
```

### 1-3. AI 자체 판단 시 명시 사항

AI가 수치/파라미터를 결정할 때 반드시:

1. **AI 판단임을 명시**
2. **그 수치 선택 이유** (학계 권장 범위, 데이터 특성, 사용자 성향 등)
3. **다른 가능한 선택지**도 간략히 제시

예시:
```
✓ "Dropout 0.3 선택. AI 판단. 근거:
   - Srivastava et al. (2014) 권장 0.2~0.5의 중간값
   - 우리 데이터 부족 (1:21) → 보수적으로 큰 값
   - 대안: 0.5 (강한 정규화) 또는 0.2 (약한 정규화)"
```

## 2. 코드 작업 시 원칙

### 2-1. 사용자 코드 수정 시

- **사용자 코드를 부분 수정**. 전체 재작성 금지.
- **필요한 부분만 손대기**. 관련 없는 부분 건드리지 말 것.
- 수정 전후 차이 명확히 설명

### 2-2. 새 코드 작성 시

- **불필요한 연산 제거**. 효율성 검토 필수.
- 변수명/함수명은 ARCHITECTURE.md와 일관
- Look-ahead bias 가능성을 항상 의심
- Forward fill만 (backward fill 절대 금지)

### 2-3. 모델 파라미터 결정

각 ML 모델의 **튜닝 가능한 파라미터**를 명시하고, 사용자에게 조절 옵션 제시:

#### LSTM (PyTorch)
```python
LSTMConfig(
    seq_len=60,        # 입력 시퀀스 길이 (논문: 60~240)
    hidden1=64,        # ★ 학습 효율: 32(언더핏)/64(균형)/128(과적합 위험)
    hidden2=32,        # ★ hidden1의 절반 권장
    dropout=0.3,       # ★ 0.2(약)/0.3(중)/0.5(강)
    lr=1e-3,           # 0.001(표준)/5e-4(보수)/1e-3(공격)
    batch_size=32,     # ★ 16~64
    max_epochs=100,    # early stopping 있음
    es_patience=10,    # ★ 검증 손실 정체 epoch
    weight_decay=0,    # L2 regularization (추가 검토 가능)
)
```

#### HMM (hmmlearn)
```python
HMMConfig(
    n_components=3,           # ★ 2~4 (BIC로 최적화 가능)
    covariance_type="full",   # ★ "full"/"diag"/"tied"
                              #    데이터 적으면 "diag"
    n_iter=100,               # EM 반복
    tol=1e-4,                 # 수렴 기준
    random_state=42,          # 재현성
)
```

#### LightGBM (현재 미사용, 향후 가능)
```python
LGBMClassifier(
    n_estimators=100,         # ★ 학습 횟수 (50~500)
    max_depth=-1,             # ★ -1(무제한)/6/8/10
    num_leaves=31,            # ★ 2^max_depth보다 작게
    learning_rate=0.05,       # ★ 0.01(보수)/0.05/0.1
    min_data_in_leaf=20,      # ★ 과적합 방지 (큰 값일수록 보수적)
    feature_fraction=0.8,     # ★ feature subsample
    bagging_fraction=0.8,     # ★ sample subsample
    reg_alpha=0.1,            # ★ L1 regularization
    reg_lambda=0.1,           # ★ L2 regularization
    early_stopping_rounds=20, # 검증 성능 정체 시 중단
)
```

**★ 표시**: 학습 효율에 직접 영향. 사용자 요청 시 조절 가능.

### 2-4. 백테스트 검증 필수 항목

새 코드 작성/수정 후 반드시 검증:

1. **비중 합 = 1** (각 시점)
2. **음수 비중 없음**
3. **NaN 없음**
4. **거래일 인덱스 정합**
5. **단위 테스트**: engine.py의 4가지 테스트 케이스 통과

## 3. 거시경제 비중에 관한 원칙

UCHIDA는 거시 자산배분 시스템. 다음 원칙 절대 준수:

### 3-1. 비중 결정 원칙

- **단일 자산 100% 금지** (분산 의무)
- **레버리지 금지** (max_leverage = 1.0)
- **Short 금지** (모든 비중 ≥ 0)
- **현금성 자산 (SOFR) 최소 0%**: 위기 외엔 강제 보유 안 함

### 3-2. 국면별 비중 원칙

| 국면 | 주식 비중 | 위기 자산 | 비고 |
|---|---|---|---|
| ATTACK | 80~90% | 10~20% | 성장 추구 |
| DEFENSE | 40~60% | 40~60% | 균형 |
| CRISIS | 0~20% | 80~100% | 자본 보존 |

### 3-3. 매크로 신호 변경 시

5개 핵심 신호 (CPI, 신용 스프레드, 장단기 역전, VIX, MA200)는 학계 검증된 거시 지표.
**제거하거나 가중치 크게 바꿀 때는 학계 근거 필수**.

신호 추가는 가능. 단, 학계 검증된 것만:
- ADX (Wilder 1978)
- Bollinger Band squeeze
- 시장 모멘텀 (Jegadeesh & Titman 1993)

## 4. 자산 운용 시 원칙

### 4-1. 실전 매매 원칙

- 백테스트 = 미국 ticker (QQQ, VYM 등)
- 실전 매수 = 한국 ETF (config.ASSETS의 code_kr)
- 환율 변동은 별도 추적 권장

### 4-2. ISA 운용 원칙

- 연간 납입한도 2,000만원 추적
- 비과세 한도 400만원 (서민형)
- 의무 가입기간 3년 (중도 해지 시 일반 과세)
- 분배금도 비과세 한도에 포함됨

### 4-3. 위기 대응 원칙

- 신호 발동 시에도 **5% 밴드** 적용 (과민 반응 방지)
- 위기 진입 시: 며칠 lookback 후 확인되면 거래
- 위기 해제 시: 더 천천히 (비대칭 유예)
- 패닉 매도 금지: 시스템 신호 신뢰

### 4-4. 적립금 처리

- 월 입금액은 그 시점 target_weights에 따라 자동 배분
- 추가 입금이 5% 밴드를 트리거하지 않도록 주의
- ISA 한도 초과 입금 경고

## 5. 학술 연구 원칙 (학교 수업 연계)

### 5-1. 학교 수업과 연결

- ECO4126 "인공지능과 금융공학" (김주철 교수, Deep Hedging 기반)
- Entropic Risk Measure → 평가 지표로 사용
- pfhedge 직접 사용 X (개념 차용)
- 학교 방식이 모델을 막으면 즉시 알려줄 것

### 5-2. 학술적 정직성

- 발견한 alpha가 작아도 사실대로 보고
- 시스템이 QQQ를 못 이기면 그렇게 말함
- 백테스트 결과 과장 금지

## 6. 협업 모드

### 6-1. AI는 다음 작업을 명시

매 응답 끝에 다음 작업의 적합 모델 표시:

- **Sonnet**: 코드 작성, 구현, 파일 생성, 검증, 단순 분석
- **Opus**: 복잡한 설계 결정, 학술적 판단, trade-off 분석, 전략 결정

### 6-2. 파일 위치 명시

파일을 줄 때마다 반드시 위치 명시:

```
📁 uchida_v3/models/baseline.py
```

### 6-3. 사용자 선호 (확정)

- **맹목적 칭찬 금지** — 비판적 사고 우선
- **출처 명시** — 논문 + 영향력 있는 인물
- **AI 판단 명시** — 변수/조절 변수/수치 결정 시 사유 필수
- **코드 부분 수정** — 새로 짜지 말고 필요 부분만
- **불필요한 연산 점검** — 효율성 항상 확인

### 6-4. 질문 패턴

- 사용자가 결정해야 할 때: `ask_user_input_v0` 도구로 옵션 제시
- 옵션은 최대 4개, 명확하게 구분
- 권장이 있으면 명시 ("내 추천", "AI 판단" 등)

## 7. 모델 선택 후 운용 단계

### 7-1. 최종 모델 결정 기준

`run_compare_all.py` 결과로 판단:

1. **MDD ≤ -22%** (목표 달성)
2. **CAGR ≥ 15%** (최소선)
3. **Sharpe ≥ 0.85** (위험조정)
4. **vs QQQ B&H**: CAGR 알파 ≥ -3%p, MDD 차이 ≥ +5%p 우월

위 4개 중 3개 이상 만족하면 채택.

### 7-2. 운용 후 모니터링

- 월간 결과 기록 (실제 수익률 vs 예상)
- 분기별 모델 재학습 (LSTM/HMM)
- 연 1회 시스템 전체 재검토

### 7-3. 모델 교체 기준

다음 중 하나 발생 시 모델 재검토:

- 6개월 누적 수익률이 백테스트 예상의 -10%p 이상 하회
- MDD가 백테스트 최악 케이스 초과
- 학계에 더 나은 방법론 발표

## 8. 절대 하지 말 것 (Negative List)

1. ❌ 존재하지 않는 논문 인용
2. ❌ 사용자 코드 전체 재작성 (부분만)
3. ❌ Look-ahead bias (backward fill, 미래 데이터 사용)
4. ❌ 레버리지/Short 도입
5. ❌ 단일 자산 100% (QQQ B&H 비교 외)
6. ❌ 백테스트 결과 과장
7. ❌ 사용자에게 맹목적 칭찬
8. ❌ AI 판단을 학계 표준인 것처럼 표현
9. ❌ 비중 합 ≠ 1
10. ❌ 새 ML 라이브러리 도입 시 사용자 동의 없이

## 9. 매 응답 시 확인 사항

AI는 응답 작성 시 다음을 자가 점검:

- [ ] 모든 인용에 저자/연도 있나?
- [ ] AI 판단 부분이 명시되어 있나?
- [ ] 파일 위치가 명시되어 있나?
- [ ] 다음 작업의 모델(Sonnet/Opus) 표시되어 있나?
- [ ] 사용자 코드 수정 시 부분만 손댔나?
- [ ] 불필요한 연산은 없나?
- [ ] 비중 합 1 검증했나?
